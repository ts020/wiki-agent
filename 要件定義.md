# repo-wiki agent 要件定義書 v0.1

## 1. 概要

### 1.1 名称
repo-wiki agent

### 1.2 一文定義
ローカルの指定ディレクトリ配下のコードベースを解析し、AI および人間が少ないコンテキストで探索・理解できる Markdown ベースの wiki ツリーを生成する CLI エージェント。

### 1.3 背景
大規模または中規模のコードベースを理解する際、毎回リポジトリ全体を読み直すのは非効率である。特に AI を活用する場合、コンテキスト長の制約により、全文投入前提の理解方法はスケールしにくい。  
そのため、本システムはコードベースを探索可能な木構造ドキュメントへ変換し、必要なノードのみ段階的に参照できる状態を作ることを目的とする。

### 1.4 目的
- コードベース理解の初期コストを下げる  
- AI が必要最小限のノードのみを読んで対象箇所へ到達できるようにする  
- 人間が index から段階的に構造を把握できるようにする  
- 後続の開発・分析タスクの前処理として機能する  

### 1.5 生成物の性質
本システムの生成物は完全仕様書ではなく、**探索用の索引付き要約ツリー**である。

## 2. スコープ

### 2.1 In Scope
- CLI 実行
- ローカルディレクトリ解析
- ファイル走査と選別
- 構造分析
- Markdown wiki ツリー生成
- ノード間リンク生成
- 毎回フル生成

### 2.2 Out of Scope
- 差分更新
- IDE 連携
- 自動コード修正
- 実行時解析
- UML / 図生成
- 完全な解析保証

### 副次利用者
- コード理解を必要とする AI
- 技術調査者

## 4. 用語定義

- **ノード**: 1つの Markdown ドキュメント  
- **wiki ツリー**: ノードの集合とリンク構造  
- **主要ファイル**: 説明の根拠となるファイル  
- **関連ノード**: 次に参照すべきノード  
- **フル生成**: 毎回すべて再生成  

---

## 5. ユースケース

### UC-01 オンボーディング
index.md から全体像を把握する

### UC-02 AI探索
必要ノードのみ読んで対象箇所へ到達する

### UC-03 ディレクトリ理解
特定ディレクトリの責務と構造を把握する

### UC-04 開発フロー理解
セットアップやテスト方法を理解する

---

## 6. 前提

- CLI アプリケーションとして動作
- 入力はローカルディレクトリ
- 出力は Markdown
- 非破壊操作

---

## 7. 機能要件

### FR-01 入力
ディレクトリ未指定時はカレントディレクトリを使用する

### FR-02 出力
デフォルト出力先は `./repo-wiki`

### FR-03 フル生成
毎回全ノードを再生成する

### FR-04 走査
ディレクトリを再帰的に走査する

### FR-05 除外
以下を除外する：
- `.git`
- `node_modules`
- `dist`
- `build`
- `target`

### FR-06 構造抽出
以下を抽出する：
- ディレクトリ構造
- 技術スタック
- エントリポイント
- テスト構造

### FR-07 ノード分割
単一文書ではなく複数ノードに分割する

### FR-08 index.md
全体概要と導線を持つ

### FR-09 ノード構成
各ノードは以下を含む：
- Summary
- Key files
- Responsibilities
- Related
- Read next

### FR-10 Markdown出力
プレーン Markdown 形式で出力

---

## 8. 非機能要件

- 可読性（短時間で理解可能）
- コンテキスト効率（最小読解）
- 冗長性抑制
- 拡張性（差分更新など）
- 非破壊性
- 再現性

---

## 9. 出力要件

### 構造

repo-wiki/
├── index.md
├── overview/
├── directories/
└── development/

### index.md
- 全体説明
- 技術スタック
- ノード導線

### ノード制約
- 短い
- 自己完結
- 次の導線を持つ

---

## 10. 制約

- ローカル実行
- Markdown 出力
- コード非変更

---

## 11. エラー処理

- 読めないファイルは無視して継続
- 生成失敗ノードは簡易出力
- 出力失敗時はエラー

---

## 12. 受け入れ基準

- index.md が生成される
- 複数ノードが生成される
- ノード構造が存在する
- 除外対象が無視される
- 任意ノード単体で理解可能
- コードベースが変更されない
