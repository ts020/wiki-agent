# なぜ repo-wiki-agent を作るのか

## 解く問題

コードベースに対して「次にどこを読めばいいか」を段階的に教えてくれる索引が存在しない。

全部読むか、何も読まないかの二択しかない。

## 既存ツールの分類と問題

### コード丸投げ系（Repomix, Code2Prompt）

コードをそのまま1ファイルに束ねてLLMに渡す。

- 構造分析も要約もしない — ただのコピペ
- 大規模リポジトリでトークン制限を超える
- 生成物は使い捨てのプロンプト素材であって、残るドキュメントではない

### シンボルマップ系（Aider RepoMap, Serena, RepoMapper）

tree-sitter や LSP でシンボル（関数名、クラス名）を抽出し、構造マップを作る。

- シンボルの「存在」はわかるが「意味」や「なぜそうなっているか」はわからない
- AIツール内部での利用に閉じていて、独立したドキュメントとして出力されない
- 人間が読んで理解を深める用途には向かない
- LSP/tree-sitter 対応言語に縛られる

### Wiki/ドキュメント生成系（DeepWiki, Google Code Wiki, CodeWiki, deepwiki-rs）

AIでコードベースからwikiやドキュメントを自動生成する。最もコンセプトが近い。

- DeepWiki — SaaS前提。パブリックリポジトリのみ。ローカル完結しない
- Google Code Wiki — Google管理のSaaS。ローカル実行不可
- CodeWiki — 学術研究。マルチエージェント構成で重く、CLIツールとして実用的でない
- deepwiki-rs — C4アーキテクチャ図に特化。汎用的な探索用wikiツリーとは目的が違う

## 何を作らないか

| 作らないもの | 理由 |
|---|---|
| コードを束ねるだけのツール | Repomix/Code2Promptが既にある。LLMに丸投げはスケールしない |
| シンボルマップ/LSP連携 | Serena/Aiderが既にある。シンボル一覧では構造は理解できない |
| SaaS/Webサービス | DeepWiki/Google Code Wikiが既にある。ローカル完結でないと私的リポジトリに使えない |
| C4アーキテクチャ図 | deepwiki-rsが既にある。図より「次にどこを読むか」の導線が重要 |
| 完全な仕様書 | 網羅性を追うと生成コストが爆発し、読む側も全部読む羽目になる |
| リアルタイム解析ツール | IDEやAIツールの中で動く必要はない。生成して置いておくもの |

## このプロジェクトが埋める空白

**ローカル完結のCLIで、段階的に探索可能なMarkdown wikiツリーを生成するツールが存在しない。**

- 丸投げ系 → 構造理解がない
- シンボル系 → 意味の説明がない
- Wiki生成系 → ローカルで動かない

この3つの穴をすべて塞ぐのが repo-wiki-agent の位置づけ。
