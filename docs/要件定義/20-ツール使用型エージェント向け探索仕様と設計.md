# 20. ツール使用型エージェント向け探索仕様と設計

本章は、Qwen3 のようなツール使用型 LLM エージェントが、128k 程度のコンテキストでも `md-wiki` の生成物を使って agentic search を行えるようにするための仕様と詳細設計である。

[18. 巨大 Markdown 対応 完了条件](18-巨大Markdown対応完了条件.md) は「巨大 Markdown を page budget 内の小ページ群へ変換できたか」を定義する。[19. 巨大 Markdown 対応 設計と機械検証](19-巨大Markdown対応設計と機械検証.md) はその実装部品と gate を定義する。本章はさらに上位の利用契約として、「エージェントがどの入口から入り、どの索引を使い、どのページを読み、どの metadata で根拠を追跡するか」を固定する。

## 20.1 ゴール

ツール使用型エージェント向け探索仕様のゴールは、LLM に生成 wiki 全体を読ませることではない。ローカルファイル操作ツールを持つ agent が、必要な小ページだけを段階的に読み、根拠ページと元 Markdown 範囲を追跡しながら回答・編集・調査を行える状態にすることである。

完了状態では以下を満たす。

- 128k コンテキストの agent が root から探索を開始できる
- agent が `list files`、`read file`、`search text`、`follow Markdown link` 相当の操作だけで目的ページに到達できる
- 生成ページ 1 枚は hard limit 40,000 文字以内に収まる
- root / agent guide / 階層 index / heading index / link index / tag index / page catalog を辿れば、全文を読まずに候補を絞れる
- entry / shell / leaf / paged index は機械可読 metadata を持つ
- leaf から source file、byte range、line range を追跡できる
- 検索結果の周辺文脈は Parent / Prev / Next / Children / Backlinks で追加取得できる
- 外部 API、AI、embedding、ネットワーク、実行時データベースに依存しない
- 同一入力・同一設定では byte-identical な生成物になる

## 20.2 想定エージェント

本仕様で想定する agent は以下の制約を持つ。

| 項目 | 前提 |
|---|---|
| コンテキスト | 128k tokens 程度。実運用では tool 結果、システム指示、会話履歴も入るため、1 ページ 40,000 文字を上限とする |
| 使用ツール | ファイル一覧、ファイル読み込み、文字列検索、相対リンク先読み込み。高度な DB や embedding search は前提にしない |
| 読み方 | 全文一括読み込みではなく、索引と検索結果から候補ページを選んで読む |
| 推論 | ページ metadata、見出し、タグ、リンク、バックリンク、周辺 leaf を使って探索判断する |
| 永続状態 | agent 実行中の短期メモのみ。生成物側に探索に必要な状態を持たせる |
| 安全性 | 生成 wiki はローカル Markdown のみ。外部ネットワークや任意コード実行を要求しない |

## 20.3 Agentic Search Contract

生成 wiki は以下の contract を満たす必要がある。

| Contract | 条件 | 目的 |
|---|---|---|
| Entry contract | root `index.md` から `agent/index.md`、`fragments/_index.md`、各索引へ到達できる | agent が最初の読み込みで探索入口を把握する |
| Page budget contract | すべての `.md` が 40,000 文字以内 | 128k コンテキストで tool 結果を安全に読む |
| Metadata contract | entry / shell / leaf / paged index / agent guide / catalog page が `md_wiki` metadata を持つ | ページ種別、元範囲、ナビ、索引を機械的に判断する |
| Navigation contract | Parent / Prev / Next / Children の metadata と本文リンクが一致する | 周辺文脈を追加取得できる |
| Index contract | `fragments/_index.md`、`headings/`、`links/`、`tags/`、`_unresolved`、page catalog はページングされる | 大規模入力でも索引が読み切れる |
| Traceability contract | leaf は source file、byte range、line range を持つ | 回答根拠と原本位置を示せる |
| Link graph contract | wikilink、forward links、backlinks が page 解像度で解決される | 関連ページへ探索を広げられる |
| Determinism contract | 出力パス、metadata、索引順、ページング境界が安定する | agent の再実行と CI 検証が一致する |
| Markdown-only contract | 生成物は Markdown と YAML frontmatter の範囲に留める | 任意のローカル file tool で読める |

## 20.4 出力構造

agentic search 対応後の出力構造は、既存の `fragments/`、`headings/`、`links/`、`tags/` を維持しつつ、agent 専用の探索入口と page catalog を追加する。

```text
md-wiki/
├── index.md
├── agent/
│   ├── index.md                  # agent guide。探索手順、索引一覧、読み込み制約
│   ├── pages/
│   │   ├── index.md              # page catalog manifest
│   │   └── page-001.md           # 必要に応じてページング
│   └── terms/
│       ├── index.md              # deterministic term index manifest
│       └── page-001.md           # 見出し・タグ・ファイル名・リンクラベル由来の語彙索引
├── fragments/
│   ├── _index.md
│   └── <rel>/
│       ├── index.md
│       ├── <section>/index.md
│       └── <section>/part-001.md
├── headings/
│   ├── index.md
│   └── page-001.md
├── links/
│   ├── index.md
│   └── page-001.md
├── tags/
│   ├── index.md
│   └── <tag>.md
└── _unresolved.md
```

`agent/` 配下は Markdown のみで構成する。JSON、SQLite、embedding index は vNext の必須要件には含めない。機械判定や CI のための JSON report は gate 出力としてのみ使い、生成 wiki の常設成果物にはしない。

### `agent/index.md`

`agent/index.md` は agent が最初に読むための機械向け guide である。以下を必ず含む。

- `md_wiki.page_kind: agent_guide`
- 生成物全体の page budget と hard limit
- 利用可能な索引一覧
- 推奨探索手順
- 読み込み禁止事項または非推奨事項
- source trace の読み方
- page catalog へのリンク
- unresolved link がある場合の確認先

このページは説明的な prose を含んでよいが、先頭 metadata とリンク一覧は機械 parse 可能な形にする。

### `agent/pages/`

`agent/pages/` は全生成ページの catalog である。agent が「どのページが存在し、どの kind で、どの source 範囲を持つか」を全文検索なしで把握するために使う。

catalog の各行は Markdown table とし、以下の列を持つ。

| 列 | 内容 |
|---|---|
| `path` | 出力 root からの相対パス |
| `kind` | `entry` / `shell` / `leaf` / `paged_index` / `agent_guide` / `page_catalog` / `term_index` |
| `title` | ページ見出しまたは生成タイトル |
| `source` | source file。source 由来でないページは空 |
| `section` | ` > ` 区切りの section path |
| `chars` | 生成ページの文字数 |
| `parent` | parent path。無い場合は空 |
| `prev` | prev path。無い場合は空 |
| `next` | next path。無い場合は空 |

catalog 自体も 40,000 文字以内にページングする。`agent/pages/index.md` は全 catalog page を直接列挙するが、500 ページを超える場合は first / last / range manifest へ縮退してよい。

### `agent/terms/`

`agent/terms/` は AI や embedding を使わない deterministic term index である。対象語彙は以下に限定する。

- source file stem
- frontmatter `title`
- tags
- aliases
- h1〜h6 見出し語
- wikilink target / alias
- 通常 Markdown link label

本文全体からの自然言語キーワード抽出、要約、同義語生成、embedding 類似検索は行わない。

term index の各行は以下を持つ。

| 列 | 内容 |
|---|---|
| `term` | 正規化済み語彙 |
| `kind` | `title` / `tag` / `alias` / `heading` / `wikilink` / `link_label` / `file` |
| `pages` | 該当 entry / shell / leaf へのリンク |

日本語の分かち書きは vNext 必須にしない。日本語では見出し・タグ・リンクラベル・ファイル名をそのまま term として扱う。

## 20.5 Metadata Schema

すべての agentic search 対象ページは YAML frontmatter に `md_wiki` を持つ。

```yaml
---
md_wiki:
  schema_version: 1
  page_kind: leaf
  output_path: fragments/huge/chapter-2/topic-a/part-001.md
  title: Topic A 1
  source: docs/huge.md
  section_path:
    - Chapter 2
    - Topic A
  heading_level: 3
  split_reason: paragraph
  char_count: 28420
  byte_ranges:
    - [123456, 145678]
  line_ranges:
    - [3400, 3720]
  parent: index.md
  prev:
  next: part-002.md
  children: []
  indexes:
    headings: ../../../../headings/page-003.md
    links: ../../../../links/page-004.md
  tags:
    - design/api
  outgoing_links:
    - ../../other/index.md
  backlinks_count: 3
---
```

必須フィールドは `page_kind` により異なる。

| page_kind | 必須フィールド |
|---|---|
| `entry` | `schema_version`, `page_kind`, `output_path`, `title`, `source`, `section_path`, `char_count`, `children` |
| `shell` | `schema_version`, `page_kind`, `output_path`, `title`, `source`, `section_path`, `heading_level`, `char_count`, `parent`, `children` |
| `leaf` | `schema_version`, `page_kind`, `output_path`, `title`, `source`, `section_path`, `char_count`, `byte_ranges`, `line_ranges`, `parent` |
| `paged_index` | `schema_version`, `page_kind`, `output_path`, `title`, `char_count`, `parent`, `prev`, `next` |
| `agent_guide` | `schema_version`, `page_kind`, `output_path`, `title`, `char_count`, `children` |
| `page_catalog` | `schema_version`, `page_kind`, `output_path`, `title`, `char_count`, `parent`, `prev`, `next` |
| `term_index` | `schema_version`, `page_kind`, `output_path`, `title`, `char_count`, `parent`, `prev`, `next` |

`parent` / `prev` / `next` は、metadata と本文 blockquote nav のリンク先が一致しなければならない。存在しないリンクを指す metadata は gate failure とする。

## 20.6 Agent 探索手順

agent は以下の順で探索することを想定する。`agent/index.md` にはこの手順を短く固定文として出力する。

1. `index.md` を読む
2. `agent/index.md` を読む
3. 問いが広い場合は `fragments/_index.md` から階層を辿る
4. 問いに見出し語・固有名・識別子がある場合は `headings/` と `agent/terms/` を検索する
5. 問いにタグや分類がある場合は `tags/` を読む
6. 問いが関係性・依存・参照元を問う場合は `links/` と対象ページの `## Backlinks` を読む
7. 候補 entry / shell を読み、`children` から leaf を選ぶ
8. leaf を読み、必要なら Prev / Next を 1〜2 ページだけ追加で読む
9. 根拠が不足する場合のみ backlinks、outgoing links、同階層 sibling へ探索を広げる
10. 回答時は参照した generated page path と、metadata の source / line range を根拠として保持する

agent は root から全 leaf を順番に読むべきではない。catalog は候補を絞るために使い、本文読み込みは leaf 単位で行う。

## 20.7 Query Routing

agent guide は、問い合わせ種別ごとの一次索引を明示する。

| Query type | 例 | Primary route | Secondary route |
|---|---|---|---|
| 定義・用語 | 「PagePlan とは」 | `agent/terms/`、`headings/` | `fragments/_index.md` |
| 仕様確認 | 「40,000 文字制限はどこで定義」 | `headings/`、文字列検索 | 関連 leaf の Prev / Next |
| 関係性 | 「Backlinks と Related の違い」 | `links/`、対象 leaf の Backlinks | 設計判断ページ |
| タグ分類 | 「auth/session 関連」 | `tags/` | `agent/terms/` |
| 未解決調査 | 「解決できない wikilink」 | `_unresolved.md` | 参照元 page catalog |
| 巨大入力範囲 | 「元ファイルのこの周辺」 | `agent/pages/` の source / line range | Prev / Next |
| 曖昧な語 | 「検索の設計」 | `agent/terms/` と全文検索を併用 | 複数候補 entry の比較 |

## 20.8 CLI 統合設計

現在の通常 CLI 経路と巨大 Markdown 経路は、最終的に `PagePlan` / `PageRegistry` で統合する。

### コンポーネント

| コンポーネント | 責務 |
|---|---|
| `InputClassifier` | scan 結果を `regular_markdown` / `large_markdown` / `skipped` に分類する。1 MiB 超の UTF-8 Markdown は skip ではなく `large_markdown` へ送る |
| `RegularPagePlanner` | 既存の h2 / h3 断片化結果を `PagePlan` へ変換する |
| `LargePagePlanner` | 19 章の section tree / forced split から `PagePlan` を作る |
| `PageRegistry` | 全 `PagePlan`、出力パス、source 範囲、リンク graph、char budget を保持する |
| `RelationPlanner` | wikilink / links / backlinks / related を page 解像度で解決する |
| `MetadataRenderer` | `md_wiki` frontmatter と本文 nav を一貫して出力する |
| `IndexPager` | `_index`、`headings`、`links`、`tags`、`_unresolved`、catalog、term index を page budget 内へ分割する |
| `AgentGuideRenderer` | `agent/index.md` と探索手順を生成する |
| `AgenticSearchGate` | 探索 fixture、query route、metadata、リンク整合性、page budget を検証する |

### 統合フロー

1. CLI が入力 path と option を受け取る
2. Scanner が Markdown 候補を収集し、NULL / invalid UTF-8 / binary を除外する
3. `InputClassifier` がファイルサイズと構造から regular / large を分類する
4. regular は既存 parser 経路で note metadata と h2 / h3 fragment を作る
5. large は streaming scanner / section tree / page planner を使う
6. 両経路の結果を `PageRegistry` に登録する
7. `RelationPlanner` が全ページの wikilink、forward link、backlink、related を解決する
8. `IndexPager` が全索引と catalog を page budget 内へ分割する
9. `MetadataRenderer` が entry / shell / leaf / paged index を出力する
10. `AgentGuideRenderer` が `agent/index.md` を出力する
11. 出力後 gate が page budget、metadata、リンク、determinism を検査する

## 20.9 入力分類ルール

vNext の入力分類は以下とする。

| 条件 | 分類 | 挙動 |
|---|---|---|
| `.md` ではない | skipped | v1 と同じく対象外 |
| NULL byte を含む | skipped | binary とみなす |
| UTF-8 として解釈不能 | skipped | warning を出す |
| 1 MiB 以下 | regular_markdown | 既存 h2 / h3 断片化を維持 |
| 1 MiB 超かつ 200 MiB 以下 | large_markdown | streaming scanner と `LargePagePlanner` を使う |
| 200 MiB 超 | large_markdown または opt-in required | 18 章の管理上限に従う |
| `fragment: false` かつ 40,000 文字超 | large_markdown | entry への全文収容ではなく hard limit を優先する。metadata に `fragment_opt_out_overridden: true` を記録する |

`fragment: false` は v1.2 では全文入口ページを許すが、agentic search hard limit では 40,000 文字超を許さない。vNext では hard limit を優先し、互換性が必要な場合は CLI option で v1.2 互換 mode を選ぶ。

## 20.10 ナビゲーション設計

本文先頭の nav は、人間にも agent にも同じ情報を提供する。

```md
> Parent: [Topic A](index.md) · Prev: [Part 1](part-001.md) · Next: [Part 3](part-003.md)
---
```

shell page は `Children` セクションを持つ。

```md
## Children

- [Overview](overview.md)
- [Part 1](part-001.md)
- [Part 2](part-002.md)
```

entry page は、その note に属する shell / leaf の一覧だけを出す。leaf が 500 を超える場合は全 leaf を列挙せず、children catalog へのリンク、first / last、range manifest を出す。

Prev / Next は同一 parent 配下で source 出現順に張る。別 parent へまたぐ Prev / Next は張らない。章をまたいだ探索は Parent へ戻って sibling を選ばせる。

## 20.11 リンク解決設計

wikilink の解決規則は既存仕様を拡張する。

| 入力 | regular markdown | large markdown / agentic search |
|---|---|---|
| `[[Note]]` | 対象 note の entry | 対象 note の entry。巨大 note では entry が children catalog を持つ |
| `[[Note#Heading]]` | h2 fragment または entry anchor | 該当 section の shell。section が leaf だけなら leaf |
| `[[Note#Heading#Sub]]` 相当 | v1 では未対応 | 見出し path 解決を導入する場合は shell / leaf へ解決 |
| 見出しが part 分割 | なし | section shell を指す。part へ直接飛ばさない |
| 解決不能 | `_unresolved.md` | `_unresolved.md`。source page と source line を記録 |

通常 Markdown link は原文を保持する。ただし local relative Markdown link が生成 wiki 内の対象へ変換可能な場合は、links index に正規化済み edge として記録してよい。

## 20.12 検証設計

agentic search 対応は `large_md_gate` に統合するか、専用 `agentic_search_gate` として実装する。いずれの場合も JSON report を唯一の機械判定入力にする。

通常 gate:

```sh
cargo run --quiet --example agentic_search_gate -- \
  --mode normal \
  --work-dir target/agentic-search-gate \
  --report target/agentic-search-gate/report.json \
  --min-score 100
```

heavy gate:

```sh
cargo run --quiet --example agentic_search_gate -- \
  --mode heavy \
  --work-dir target/agentic-search-heavy \
  --report target/agentic-search-heavy/report.json \
  --min-score 100 \
  --require-resource-budget
```

### Check ID

| Check ID | 判定式 |
|---|---|
| `agentic-entrypoint` | root `index.md` から `agent/index.md`、`fragments/_index.md`、`headings/`、`links/`、`tags/`、`_unresolved.md` へ到達できる |
| `agentic-page-budget` | 生成された全 `.md` が 40,000 文字以内 |
| `agentic-metadata` | 全対象 page の YAML frontmatter を parse でき、`page_kind` ごとの必須フィールドが存在する |
| `agentic-catalog` | `agent/pages/` に全 generated page が重複なく列挙され、path が存在する |
| `agentic-term-index` | 見出し、タグ、alias、link label の deterministic term が `agent/terms/` に出る |
| `agentic-navigation` | metadata と本文 nav の Parent / Prev / Next / Children が一致し、リンク先が存在する |
| `agentic-routing` | fixture query ごとに期待 primary route が `agent/index.md` に記載されている |
| `agentic-query-fixtures` | 代表 query が全文一括読み込みなしに期待 leaf へ到達できる |
| `agentic-traceability` | leaf から source file、byte range、line range を取得できる |
| `agentic-link-graph` | wikilink、forward link、backlink、unresolved が期待 page 解像度で一致する |
| `agentic-no-full-read` | gate の模擬 agent が source 全体または全 leaf を読む手順を必要としない |
| `agentic-determinism` | 同一 fixture を 2 回生成した output tree が byte-identical |

### Query Fixture

最低限、以下の query fixture を用意する。

| Fixture | Query | 期待 |
|---|---|---|
| heading lookup | `marker heading 314 の仕様はどこか` | `agent/terms` または `headings` から該当 shell / leaf へ到達 |
| exact phrase | `marker-no-heading-2048` | text search から該当 `part-NNN.md` へ到達し、Prev / Next は必要時だけ読む |
| relationship | `Topic A を参照しているページ` | backlinks または links index から参照元を列挙 |
| tag route | `auth/session 関連` | `tags/auth/session.md` から entry へ到達 |
| unresolved route | `解決できない wikilink` | `_unresolved.md` から source page と line range を得る |
| ambiguous heading | `Design` | 複数候補を catalog で比較し、source / section_path で絞る |
| huge single line | `single-line-marker-99999` | byte window leaf へ到達し、line range が同一行でも byte range で位置を追跡できる |

## 20.13 実装順序

実装は以下の順で進める。各項目の詳細な作業分解、依存関係、機械的な完了条件は [21. Agentic Search 実装作業計画](21-agentic-search実装作業計画.md) で定義する。

1. 本章の仕様を `docs/要件定義/index.md` に追加する
2. `agentic_search_gate` の JSON report skeleton と failing checks を作る
3. `InputClassifier` を導入し、1 MiB 超 `.md` を skip ではなく large path へ分類する
4. 既存 regular fragment を `PagePlan` に変換する adapter を作る
5. `PageRegistry` を導入し、regular / large の出力計画を統合する
6. `MetadataRenderer` を導入し、entry / shell / leaf / paged index に `md_wiki` frontmatter を出す
7. `IndexPager` を通常 CLI 出力にも適用する
8. `agent/index.md`、`agent/pages/`、`agent/terms/` を生成する
9. wikilink / links / backlinks を shell / leaf 解像度へ拡張する
10. normal gate を `scripts/verify.sh` に追加する
11. heavy gate を明示実行 CI job として追加する
12. v1.2 互換 mode が必要な場合だけ、soft limit warn の旧挙動を option 化する

各段階は AC-01〜AC-23 の既存テスト、品質スコア、large Markdown gate を落とさないことを前提とする。

## 20.14 現状との差分

現時点の実装は、通常サイズの構造化 Markdown では `fragments/_index.md`、`headings/`、`links/`、`tags/`、Parent / Prev / Next による探索が可能である。一方、agentic search 完了状態とは以下の差分がある。

| 項目 | 現状 | 完了状態 |
|---|---|---|
| 通常 CLI と巨大 Markdown | 別経路 | `PageRegistry` で統合 |
| 1 MiB 超入力 | scan で skip | large path へ分類 |
| 40,000 文字超 | warn して継続 | gate failure。生成物は超過 0 |
| metadata | large gate の leaf 中心 | 全 page kind に `md_wiki` metadata |
| shell / paged index | 一部仕様のみ | 実出力で必須 |
| agent guide | なし | `agent/index.md` を生成 |
| page catalog | なし | `agent/pages/` を生成 |
| deterministic term index | なし | `agent/terms/` を生成 |
| query fixture | なし | 模擬 agent 探索 gate で検証 |

したがって、現状は「通常サイズの Markdown wiki を agent が辿れる」段階であり、本章の完了状態は「巨大 Markdown を含む入力でも、128k コンテキストの tool-use agent が全文一括読み込みなしに根拠 leaf へ到達できる」段階である。
