# 8. 機能要件

## 表記ルール（前提）

本書では「断片」と「断片ページ」を下表のように使い分ける。

| 語 | 意味 |
|---|---|
| 断片 | ノートから切り出した論理的なかたまり（h2 または h3 のスコープ） |
| 断片ページ | 断片を実体化した `.md` ファイル（h2 断片または h3 子断片） |
| 入口ページ | ノートに 1 対 1 で対応する `fragments/<rel>/index.md` |
| 殻ページ | h3 再分割された h2 に対応する `fragments/<rel>/<h2-slug>/index.md` |

以下、文中の「ページ」は入口・殻・断片の総称。

## FR-01 CLI エントリ
`md-wiki <INPUT> [-r] [-o <DIR>]` として動作する。入力指定と出力指定の詳細は [§7](07-入力.md) に従う。

## FR-02 走査
- 入力がファイルなら当該ファイル 1 枚のみを対象とする
- 入力がディレクトリかつ `--recursive` なしなら、直下の `.md` のみ
- 入力がディレクトリかつ `--recursive` ありなら、配下を再帰走査

[§7.4 除外対象](07-入力.md#74-除外対象) と [§7.5 走査ガード](07-入力.md#75-走査ガード) に従う。

## FR-03 フル生成
毎回すべてのノート・断片・索引を再生成する。差分更新は v1 スコープ外（[§14](14-将来拡張.md)）。出力先ディレクトリに既存の生成物がある場合は、実行前にクリーンアップしてから書き直す（マージはしない）。

## FR-04 ノート取り込み
[§7.2](07-入力.md#72-取り込み判定) のルールで対象 md を選別する。

## FR-05 ノート出力（断片化）
取り込んだ `.md` は、**入口ページ + 断片ページ群** として `fragments/<入力からの相対パス（拡張子を除いたもの）>/` 配下に展開する。

ネスト例: 入力 `docs/auth/session.md` → 入口 `fragments/docs/auth/session/index.md`。

### 5.1 断片化トリガ

以下のいずれかに該当するノートは**非断片化**とし、入口ページに全文（プレフェイス＋本文すべて）を収める。断片ページは 1 つも生成しない。

- 本文に h2 が 1 つも無い（h3 以下のみ／プレーンテキストのみのノート）
- フロントマターに `fragment: false` がある

上記以外は断片化を行う（h2 が 1 つのノートでも、入口ページ ＋ 断片ページ 1 枚に分ける）。

### 5.2 断片化のアルゴリズム

1. ノート先頭から最初の h2 直前までを **プレフェイス** として切り出す（h1 とその直下の本文、表、リスト等）
2. 残りの本文を h2 境界で分割し、各 h2 セクションを **断片** とする
3. 各 h2 セクションの本文長（後述 5.3）が **300 行超**、かつ当該 h2 配下に h3 見出しが **2 個以上**存在するとき、その h2 は**殻**に降格し、各 h3 セクションを子断片として分割する
4. h3 より深い見出しは分割の単位にしない（断片ページ内部に原文のまま残す）

### 5.3 行数の数え方

3 の閾値判定に用いる「本文長」は、当該 h2 見出しの**直後の行**から、**次の h2 の直前の行**までの行数を **LF 単位**で数える（CRLF は LF 正規化後にカウント）。

- 空行・フェンスドコードブロック内の行・h3 以下の見出し行を**すべて含めて**数える
- h2 見出し行そのものは**含めない**
- 比較は「本文長 > 300」で判定する（300 行ちょうどは断片のまま）

### 5.4 出力物

- **入口ページ** `fragments/<rel>/index.md`
  - 原本のプレフェイス（h1 と本文冒頭）をそのまま含む
  - 本文末尾に `## Fragments` セクションを生成し、各直下断片（h2 通常断片 or 殻ページ）へのリンクを原本出現順に列挙する（断片が 0 件なら省略）
  - 末尾に `## Backlinks` / `## Related`（FR-10, FR-11）を付与する
- **断片ページ（h2 通常）** `fragments/<rel>/<h2-slug>.md`
  - ページ冒頭にナビゲーション（FR-05a）
  - 続けて h2 見出しとその本文（h3 以下を含む）
  - 末尾に `## Backlinks`（FR-10）。Related は付与しない（FR-11）
- **殻ページ** `fragments/<rel>/<h2-slug>/index.md`（h3 再分割時のみ）
  - 冒頭ナビは Parent のみ（Prev / Next は出さない）
  - 当該 h2 のプレフェイス（h2 見出し直後から最初の h3 手前まで）を配置
  - 続けて `## Fragments` で子断片一覧
  - 末尾に `## Backlinks`。Related は付与しない
- **子断片ページ（h3）** `fragments/<rel>/<h2-slug>/<h3-slug>.md`
  - 冒頭ナビ（FR-05a）＋ h3 見出しと本文
  - 末尾に `## Backlinks`。Related は付与しない

### 5.5 本文の加工

- `[[wikilink]]` は標準 Markdown リンクへ変換する（FR-06）
- 原本が持つ見出しレベルは変えない（断片ページに切り出しても h2 は h2 のまま）
- 自動セクション（Backlinks / Related / Fragments / ナビ）と原本本文との間は水平線（`---`）で区切る

### 5.6 スラグと衝突

- 断片 slug は FR-08 の見出しスラグ仕様に従う
- **同一ノート内**で h2 slug どうしが衝突したら出現順に `-1`, `-2`, ... を付与する
- **同一ノート内**で「殻化する h2 の slug」と「別の h2 通常断片の slug」が衝突した場合は、後者も slug 衝突とみなし `-1` を付与する（ディレクトリとファイルが同名で共存するのを避けるため、ファイル名側を改名する）
- h3 再分割時の子断片 slug 衝突は当該 h2 内でのみ解決する（別の h2 配下の slug とは干渉しない）
- **入力ディレクトリ側の衝突**: 同じ階層に `foo.md` と `foo/` サブディレクトリが共存する場合、前者の入口は通常通り `fragments/.../foo/index.md`、後者配下のノートは `fragments/.../foo/<...>/index.md` となる。もし `foo.md` 由来の h2 slug が、`foo/` 配下のノート名（basename）と衝突したら、h2 側の slug に `-1` を付与して回避する

## FR-05a 断片ナビゲーション

**h2 通常断片ページ** と **h3 子断片ページ** の冒頭に、以下のナビ行を 1 行（blockquote）で挿入し、直後に水平線（`---`）を置く。殻ページは Parent のみを同形式で出す。入口ページにはナビを付けない。

- **Parent**: 1 つ上のページへのリンク。h2 通常断片 / 殻ページは入口ページへ、h3 子断片は殻ページへ
- **Prev**: 同階層で原本出現順の 1 つ前の断片。無ければ項目ごと省略
- **Next**: 同階層で原本出現順の 1 つ後の断片。無ければ項目ごと省略

ナビ行フォーマット（項目が無いときは項目と前後の ` · ` を削る）:

```
> Parent: [<note title>](../index.md) · Prev: [<prev heading>](prev.md) · Next: [<next heading>](next.md)
```

例:

- 先頭の h2 断片（Prev 無し）: `> Parent: [...](index.md) · Next: [...](next.md)`
- 末尾の h2 断片（Next 無し）: `> Parent: [...](index.md) · Prev: [...](prev.md)`
- 唯一の h2 断片（Prev/Next 共に無し）: `> Parent: [...](index.md)`
- 殻ページ: `> Parent: [...](../index.md)`

## FR-06 wikilink 解釈
Obsidian 互換の `[[...]]` 記法を解釈し、出力では相対パスの Markdown リンクへ変換する。

### 対応構文

| 構文 | 意味 | 出力 |
|---|---|---|
| `[[NoteName]]` | ノート参照 | `[NoteName](<入口ページへの相対パス>)` |
| `[[NoteName\|表示名]]` | エイリアス付き | `[表示名](<入口ページへの相対パス>)` |
| `[[NoteName#見出し]]` | h2 アンカー | `[NoteName#見出し](<対応する h2 断片または殻ページへの相対パス>)` |
| `![[NoteName]]` | 埋め込み | v1 未対応。`[[NoteName]]` と同じ扱い（プレーンリンクに縮退、`!` 前置は削除） |

### `#見出し` の解決

- 対象ノート内の **h2** スラグと完全一致した場合のみ成功する
- 成功時、対象が通常断片なら断片ページ、殻化されていれば殻ページの `index.md` を指す
- h1 / h3 以降との一致は成立させず **未解決**（h1 への参照は `[[NoteName]]` で入口を指すため十分）
- 対象ノートが非断片化（FR-05 5.1）なら、`#見出し` 指定はノート名部分だけ解決し、アンカーは `#<見出しスラグ>` として入口ページ内に追加する

### 解釈対象外の領域

以下に含まれる `[[...]]` は解釈せず原文保持：

- フェンスドコードブロック内（``` または ~~~ で囲まれた領域）
- インラインコード内（`` ` `` で囲まれた範囲）
- HTML コメント内（`<!-- ... -->`）

### 解決規則

ターゲット名の候補を以下の優先順で選ぶ。上位で 1 件に確定しない場合のみ下位のルールに進む：

1. `aliases` / `title` に完全一致するノート
2. 出力先 basename（拡張子除く）に完全一致するノート
3. （ここまでで複数残ったとき）参照元と同一ディレクトリ配下のもの
4. （それでも複数残ったとき）最短パスのもの
5. （それでも複数残ったとき）相対パス文字列のアルファベット順

すべて空集合、または `#見出し` が対象ノートに見つからない場合は **未解決** とみなす。未解決時の本文表記は FR-13 を参照。

## FR-07 タグ索引

- フロントマターの `tags: [a, b]` を第一級タグとして扱う
- ネストタグ `auth/session` は **`tags/auth.md`** と **`tags/auth/session.md`** の双方に集計する
  - つまり、親タグのページ `tags/auth.md` は、`auth` 自身を付けたノートも、`auth/session` のような任意の子孫タグを付けたノートも包含する（Obsidian 互換）
- `tags/<tag>.md` は自動生成し、該当ノートの**入口ページ**をフラットに列挙
- `tags/index.md` を生成し、全タグを列挙した入口ページとする
- **親子衝突**: 親タグページは常に `tags/<tag>.md` として単一ファイルで生成する。子タグは `tags/<tag>/<child>.md` にぶら下げる。同名のファイルとディレクトリ（例: `tags/auth.md` と `tags/auth/`）はほとんどの FS で共存可能なので、衝突回避の特別処理は入れない
- 本文内の `#tag` 記法は v1 では解釈しない（コードスニペット等との誤検出回避）

## FR-08 見出し索引
全ノートの h1〜h2 を横断集約した索引を生成する。

- 出力先: `headings/index.md`
- 各エントリは「見出しテキスト / 所属ノート名 / リンク先」を持つ
- h1 のリンク先は当該ノートの **入口ページ** (`fragments/<rel>/index.md`)
- h2 のリンク先は対応する **断片ページ** (`fragments/<rel>/<h2-slug>.md`、h3 再分割時は `fragments/<rel>/<h2-slug>/index.md`)
- 非断片化ノート（FR-05 5.1）の h2 は、入口ページ内のアンカー `fragments/<rel>/index.md#<h2-slug>` を指す
- 断片ページ化によりページ先頭が見出しそのものとなるため、通常の h2 リンクに `#slug` アンカーは付けない
- h3 以上は v1 では対象外（[§14](14-将来拡張.md)）

### 見出しスラグ仕様（再現性のため固定する）

1. Unicode NFKC 正規化
2. ASCII 英大文字は小文字化
3. 空白（`\s+`）は単一の `-` に置換
4. 次の文字は除去: 句読点 `` .,:;!?`'"()[]{}<>@#$%^&*+=|\/~ `` と制御文字
5. 残った連続 `-` は 1 個に畳み、先頭末尾の `-` を除去
6. 日本語・CJK・記号等は NFKC 後にそのまま残す（GitHub 風挙動）
7. 同一ノート内で衝突したら、2 番目以降に `-1`, `-2`, ... を付与（出現順）

## FR-09 リンク索引
断片ページ・入口ページの参照関係を一覧化した索引を生成する。

- 出力先: `links/index.md`
- **ソースは断片解像度**: ページ（入口・殻・断片）単位で「このページが参照しているページ」を列挙する
- ページ見出しは `### <ページタイトル>` （`タイトル` は入口ならノート title、断片なら h2/h3 見出し文字列）とする
- リンクはページタイトルと相対パス `- [<target title>](<relative path from links/index.md>)`
- 出力順は、参照元ページの出力パス辞書順 → そのページ内での参照出現順
- 逆向き（被参照）は各ページ末尾の Backlinks に出すため、リンク索引では省略する
- 未解決リンクは含めず、`_unresolved.md` に集約する

## FR-10 バックリンク

入口ページ・殻ページ・断片ページの末尾に「このページを参照している他ページ一覧」を自動付与する。

- ページ末尾の `## Backlinks` セクションに列挙
- **解像度は断片単位**: 同じノート内の別断片からのリンクも個別の参照として扱う
- 参照元が存在しなくなった場合は列に出さない（フル生成のため残らない）
- 参照が 0 件のページでは `## Backlinks` セクション自体を出さない
- 同じ参照元ページから複数回参照されている場合でも一度だけ列挙する
- 出力順は参照元ページの出力相対パス昇順（安定ソート）
- 各行の形式: `- [<参照元タイトル>](<相対パス>)`

## FR-11 Related

Related は**入口ページのみに付与**する（断片ページ・殻ページには出さない）。断片は単独で「関連ノート」を判断するための情報を持たないため、ノート単位で計算する。

- フロントマター `related` を最優先で列挙（記載順を保持）
  - 自ノートへの参照は除外
  - 既定は入口ページへのリンク
- 追加で、共通タグが 2 つ以上あるノートの**入口ページ**を推定列挙（最大 3 件）
  - スコア: 共通タグ数の降順
  - 同点時のタイブレーク: 入口ページ出力相対パスの辞書順昇順
  - `related` に既に列挙済みのノートは推定列挙でも重複させない
  - 自ノートは除外
- 各行の形式: `- [<対象タイトル>](<相対パス>)`
- 候補が 0 件なら `## Related` セクション自体を出さない

## FR-12 index.md

ルートの `index.md` は「プロジェクト全体の概況」に徹する（ノート一覧は FR-16 の `_index.md` 階層サマリに委譲）：

- プロジェクト名（入力ルートの basename、または `--out` 既定時は `md-wiki`）
- サマリ
  - **ノート数**: 取り込んだ `.md` の数（入口ページ数に等しい）
  - **断片数**: 生成した断片ページ（h2 通常 + h3 子）の総数。入口ページ・殻ページは含めない
  - **タグ数**: `tags/index.md` に列挙されるユニークなタグ名の数（親・子は別カウント）
  - **未解決数**: `_unresolved.md` に載る未解決リンクの件数
- 各セクションへの導線のみを短いリストで出す
  - `fragments/_index.md`（ノートツリーの入口。サイトマップ）
  - `tags/index.md`, `headings/index.md`, `links/index.md`, `_unresolved.md`

全ノートの列挙は行わない（[§9 コンテキスト効率](09-非機能要件.md) の 40k 文字上限を満たすため）。生成日時も含めない（冪等性を保ち、`git diff` でのノイズを避けるため）。

## FR-16 階層サマリ `_index.md`

`fragments/` 配下の各ディレクトリ（トップ `fragments/` 自身を含む）に **`_index.md`** を自動生成する。入口ページ (`index.md`) と命名衝突させないため Hugo 風に `_` を前置する。

内容（ディレクトリ `D` 配下の `_index.md`）:

1. **見出し**: `# <D の相対パス>`（トップは `# fragments`）
2. **サマリ**: 配下のノート数（再帰）・断片数（再帰）・直下サブディレクトリ数
3. **直下サブディレクトリ一覧**: 各サブディレクトリ `S` について `- [<S>](<S>/_index.md) — N ノート`（辞書順）
4. **直下ノート一覧**: `D` 直下に入口を持つノートについて `- [<note title>](<note>/index.md)`（ノート basename 辞書順）

`_index.md` は「このディレクトリへ移動した人／AI が、この階層に何があるかを 1 ページで把握する」ためのサイトマップ。ルート `index.md` → `fragments/_index.md` → 各サブディレクトリ `_index.md` と再帰的に辿れば全ノートへ到達できるため、ルートや途中の集約ページが肥大化しない。

文字数が 40,000 を超えそうなときは警告ログを出す（v1.2 では分割しない）。

## FR-13 未解決リンク
解決できなかった wikilink は `_unresolved.md` にまとめる。

- ページ単位（参照元ページ）でグループ化し、各グループ配下にターゲット名・エイリアス・見出しを一覧
- 並び順: 参照元ページの出力相対パス昇順 → グループ内で本文出現順
- ノート本文では **wikilink 記法のまま残し、直後に ` (未解決)` を付与**
  - 例: `[[Foo]] (未解決)`、`[[Foo#sec|表示]] (未解決)`
  - `![[Foo]]` が未解決となった場合も `[[Foo]] (未解決)` と縮退表記で統一
  - CommonMark パーサはこのまま素通しし、リンク化はされない
- `_unresolved.md` が空（未解決 0 件）のときも空ページを生成する（存在保証）

フォーマット例:

```markdown
# Unresolved wikilinks

## fragments/foo/index.md

- `[[Missing]]`
- `[[Foo#nope]]` — 見出し `nope` が対象ノートに存在しない
```

## FR-14 走査上限

用語:
- **走査ファイル数**: 走査で発見した `.md` の総数（スキップされた md・フロントマター `wiki: false` も含む）
- **取り込みノート数**: 実際に入口ページを生成した `.md` の数

上限:
- 走査ファイル数が 10,000 件を超えたら警告ログ（処理は継続）
- 取り込みノート数が 5,000 件を超えたら警告ログ（処理は継続）
- ディレクトリ深さが 20 段を超えたら警告ログ（処理は継続）
- 個別ファイルの上限は [§7.5](07-入力.md#75-走査ガード)

## FR-15 増築性
入力ディレクトリに `.md` を追加して再実行するだけで、以下がすべて自動更新される：

- `fragments/` への入口・断片追加
- タグ索引の追記
- 見出し索引の追記
- リンク索引の追記
- バックリンク
- 未解決リンクの解決（追加したノートが既存未解決の対象だった場合）
