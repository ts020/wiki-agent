# 8. 機能要件

## FR-01 入力
ディレクトリ未指定時はカレントディレクトリを使用する。

## FR-02 出力
デフォルト出力先は `./repo-wiki`。

## FR-03 フル生成
毎回全ノードを再生成する。追加された手書き md は次回実行時に取り込まれる。

## FR-04 走査
ディレクトリを再帰的に走査する。

## FR-05 除外
以下を除外する：
- `.git`
- `node_modules`
- `dist`
- `build`
- `target`

## FR-06 構造抽出（静的解析ベース）
以下をルールベースで抽出する：
- **ディレクトリ構造**: 走査結果から木構造を生成
- **技術スタック**: プロジェクトファイル（`Cargo.toml`, `package.json`, `pyproject.toml`, `requirements.txt`, `go.mod`, `pom.xml`, `Gemfile` 等）をパース
- **エントリポイント**: 言語別ヒューリスティック（`main.rs`, `src/main.*`, `index.ts`, `__main__.py` 等）
- **テスト構造**: 命名規約とディレクトリ規約（`tests/`, `__tests__/`, `*_test.rs`, `*.test.ts`, `test_*.py` 等）

### v1 対応言語（シンボル抽出）
以下 4 言語は正規表現ベースで公開シンボル（関数・型）を抽出する。対応外言語はファイル名列挙のみ。
- Rust: `fn`, `struct`, `enum`, `trait`, `pub` 修飾子の有無で公開判定
- TypeScript / JavaScript: `export` 付きの `function` / `class` / `interface` / `type` / `const`
- Python: モジュール直下の `def` / `class`。`_` 始まりは非公開とみなす
- Go: 大文字始まりで公開判定。`func`, `type`

## FR-07 ノード分割とサイズ制約
- **コード由来**: ディレクトリ単位で 1 ノード生成
- **ノート由来**: 手書き md 1 ファイル = 1 ノード
- 単一巨大文書は生成しない

### ノード長ガイドライン
- ソフト目標: 100 行以下
- ハード上限: 300 行。超過時は以下で緩和：
  - サブディレクトリへの分割（コード由来）
  - Structure 欄の一部を別ノード（後述）に退避
- コード由来ノードのシンボル列挙は **1 ノードあたり最大 100 件**
  - 順序: 公開シンボル → 非公開シンボル、各グループ内はアルファベット順
  - 超過時は同ディレクトリ配下に `<dir>/_symbols.md` を生成し、全件を移す。ノード本体からは相対リンクで参照
- ノート由来ノードは行数制限を課さない（ユーザ成果物を改変しないため）

## FR-08 index.md
全体概要と導線を持つ。以下を含む：
- プロジェクト名・技術スタック
- ディレクトリ別ノード一覧
- タグ索引へのリンク
- 主要手書きノートへのリンク

## FR-09 ノード構成
各ノードは以下を含む。生成方法はノード種別ごとに異なる：

| 項目 | コード由来 | ノート由来 |
|------|-----------|-----------|
| Summary | 主要ファイルから抽出したモジュール/型/関数名の要約行 | フロントマター `summary` / なければ冒頭段落 |
| Structure | 関数・型・公開シンボルの階層リスト | 見出しツリー（h1〜h3 以上） |
| Key files | ディレクトリ内の代表ファイル（README, main, index, mod.rs など） | 当該 md ファイルパス |
| Related | パス隣接ノード + 参照先・参照元 | フロントマター `related` + 自動リンク解析結果 |
| Read next | 同階層・子ノードの優先順リスト | タグ共通・参照関係から推定 |

## FR-10 Markdown 出力
CommonMark 準拠のプレーン Markdown で出力。Obsidian 互換の `[[wikilink]]` 記法を解釈・変換する（詳細は FR-11）。

## FR-11 wikilink 解釈
Obsidian 準拠の記法を取り込み md 内で解釈し、出力では相対パスの標準 Markdown リンクへ変換する。対象は **ノート由来ノードの本文** および **フロントマターの `related`**。

### 対応構文
| 構文 | 意味 | 出力 |
|------|------|------|
| `[[NoteName]]` | ノード名参照 | `[NoteName](相対パス)` |
| `[[NoteName\|表示名]]` | アリアス付き | `[表示名](相対パス)` |
| `[[NoteName#見出し]]` | 見出しアンカー | `[NoteName#見出し](相対パス#見出しのスラッグ)` |
| `![[NoteName]]` | 埋め込み（Obsidian の embed） | v1 未対応。プレーンリンクに縮退 |

### 解決規則
1. `aliases` フィールドに一致するノードを優先
2. 出力先パスの basename（拡張子除く）に完全一致
3. 候補が複数ある場合の優先順位: 同一ディレクトリ > 最短パス > アルファベット順
4. 未解決: 原文を保持しつつ末尾に ` (未解決)` を付与。index に未解決リンク一覧を出す

## FR-12 タグ
- フロントマターの `tags: [a, b]` を第一級のタグとする
- ネストタグ `auth/session` は `tags/auth.md` と `tags/auth/session.md` の双方に集計
- 本文内の `#tag` 記法は v1 では解釈しない（コードスニペット等との誤検出回避）
- `tags/<tag>.md` は自動生成。該当ノードをフラットに列挙

## FR-13 バックリンク
各ノードに「このノードを参照している他ノード一覧」を自動付与する。コード由来・ノート由来を区別して列挙する。

## FR-14 走査ガード
FR-04 の再帰走査は以下の上限と挙動を持つ：
- ファイル数上限: 10,000 件で警告ログ。処理は継続
- 生成ノード数上限: 5,000 で警告ログ
- ディレクトリ深さ: 上限なし（警告のみ: 20 段超）
- ファイルサイズ 1 MiB 超、バイナリ、シンボリックリンクは 7.1 に従ってスキップ

## FR-15 増築性
手書き md を追加してツールを再実行するだけで、以下が自動更新される：
- index の導線
- タグ索引
- バックリンク
- 関連ノード
- wikilink の解決結果（新規ノードが既存の未解決リンク先になる場合を含む）
