# 9. 非機能要件

- **小コンテキスト agent 適性**: agent が生成 wiki 全体を読まずに、索引、catalog、field、context pack から必要な根拠だけを取得できること
- **根拠追跡性**: context pack の各項目は、可能な限り generated page path と source range を追跡できること
- **retrieval 決定性**: 同一入力、同一 schema、同一 `context` command から同一 context pack が生成されること
- **透明性**: `md-wiki context` の候補は Markdown artifacts と `.md-wiki/catalog.json` から説明可能であり、LLM の不可視推論に依存しないこと
- **可読性**: 生成物は短時間で目を通せること。索引ページは「どのページから何に飛べるか」が 1 画面で把握できる規模を目安にする
- **コンテキスト効率（128k 前提）**: 生成される**個々のページ**（入口・殻・断片・索引・`_index.md`・タグ・未解決）は、**128k トークンのコンテキストに単体で収まる**こと。日本語混在を見越した実装目安は **1 ページあたり 40,000 文字（約 13k〜40k トークン）以内**。これを超える集約ページは階層的にサブページへ分割する（FR-12 / FR-16 / FR-08 / FR-09 の実装方針に反映）。超過時は警告ログを出す。通常 compile artifacts の 40,000 文字超は当面 warn とする。`context` が返す context pack は指定 budget を超えてはならない
- **部分読み込みしやすさ**: AI に食わせる時点で「ノート全体」ではなく「関連する断片＋入口のコンテキスト」だけで完結するように、断片ページは隣接ページ（Parent / Prev / Next / Backlinks）へのリンクで自己完結させる
- **再現性**: 同一入力から同一出力（ファイル順序・ディレクトリ走査順に左右されないソート）
- **冪等性**: 同じ入力で何度実行しても、出力は同じ状態に収束する
- **非破壊性**: 入力側の `.md` を変更しない
- **オフライン動作**: 外部 API / ネットワークに依存しない
- **実行コスト**: 数百〜数千件規模のノートを数秒で処理できること（目安: 1,000 ノートで 5 秒以内）
- **拡張性**: `add` の差分反映、watch mode、索引追加を阻害しない構造を保つ
- **ポータビリティ**: 成果物は純粋な Markdown。任意のビューア・静的サイトジェネレータ・CI で扱える
