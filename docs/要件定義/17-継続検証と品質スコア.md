# 17. 継続検証と品質スコア

本章は、v1 要件を満たしていることを継続的に確認するための検証仕様である。
`docs/要件定義/13-受け入れ基準.md` は完成条件の一覧、本章はそれを機械検証へ落とすための詳細条件と運用を定義する。

## 17.1 検証の層

検証は 3 層に分ける。

| 層 | 目的 | 失敗時の扱い | 実行手段 |
|---|---|---|---|
| Gate | v1 要件違反・回帰を止める | リリース不可 / CI 失敗 | `scripts/verify.sh` |
| Acceptance | AC-01〜AC-31 の具体シナリオを固定する | Gate に含める | `cargo test` |
| Score | 生成 wiki の品質を数値化し、コミットごとの傾向を見る | 閾値未満は Gate 失敗 | `cargo run --example quality_score` |

Gate は以下を必ず実行する。

```sh
cargo fmt --check
cargo clippy -- -D warnings
cargo test
cargo run --quiet --example quality_score -- --min-score 90
```

コミット単位でスコア履歴を残す場合は以下を実行する。

```sh
scripts/record-quality-score.sh
```

履歴は `docs/quality/history/` に `<commit>.json` と `<commit>.md` として保存する。

AC-24〜AC-31 は新規追加された retrieval compiler 受け入れ基準であり、以下の検証項目は実装タスクで追加する計画上の test target である。現時点の Gate coverage には含めない。

## 17.3 追加で固定する回帰条件

AC の短文だけでは漏れやすいため、以下も Gate の一部として扱う。

- 単一ファイル入力でも走査ガード（1 MiB 超は large path、NULL、UTF-8 不正）を通す
- frontmatter `related` は wikilink ターゲット名・title・alias・出力相対パスのいずれでも入口ページへ解決する
- `[[...]]` は fenced code block、inline code、HTML comment 内では原文保持する
- 取り込みノート数が 5,000 件を超えたら warn ログを出す
- README の出力構造説明は現行 `fragments/` 構造と矛盾しない

## 17.4 品質スコアの目的

品質スコアは「テストが通るか」だけでは見えない、生成 wiki としての読みやすさ・リンク健全性・再生成安定性を数値化する。
テストは仕様違反を止めるための Gate、スコアは出力品質の劣化を早期に見つけるための trend signal である。

スコアは固定フィクスチャを毎回生成し、出力 Markdown を検査して算出する。入力は `examples/quality_score.rs` が `target/quality-score/fixture/` に作るため、外部データやネットワークに依存しない。

## 17.5 スコア判定基準

満点は 100 点。Gate の既定閾値は 90 点とする。

| Check ID | 配点 | 判定基準 |
|---|---:|---|
| `required-files` | 15 | `index.md`, `fragments/_index.md`, `tags/index.md`, `headings/index.md`, `links/index.md`, `_unresolved.md` がすべて存在する |
| `local-links` | 20 | 生成 Markdown 内のローカル Markdown リンクがすべて存在するファイルを指す |
| `wikilink-rewrite` | 15 | 解決可能な wikilink が Markdown リンク化され、通常ページに `[[` の残骸が無く、未解決リンクが 0 件である |
| `fragment-navigation` | 15 | 入口以外の断片・殻・子断片ページが `> Parent:` で始まり、直後に `---` を持つ |
| `relations` | 15 | Backlinks と Related が期待ページに出力され、入口ページへの相対リンクが正しい |
| `context-efficiency` | 10 | 40,000 文字を超える生成ページが 0 件である |
| `determinism` | 10 | 同一入力を 2 回生成した出力ディレクトリがバイト単位で一致する |

減点は check 単位で行う。`relations` のみ Backlinks / Related の片方が成功した場合に 7.5 点とする。それ以外の check は合格なら満点、不合格なら 0 点とする。

### Retrieval checks（実装計画）

以下は retrieval compiler 実装タスクで追加予定の品質 check である。現時点では未実装であり、active な 100 点満点 Gate には含めない。実装時にスコアの再配分または満点定義を更新する。

| Check ID | 予定配点 | 判定基準 |
|---|---:|---|
| `schema-catalog` | 10 | schema 指定時に catalog と field catalog が生成され、必須項目が揃う |
| `context-pack` | 15 | game narrative fixture から valid な Markdown context pack が生成される |
| `context-budget` | 10 | context pack が指定 budget を超えない |
| `context-determinism` | 10 | 同一 context command が byte-identical |

## 17.6 スコア履歴

履歴は以下の形式で保存する。

```text
docs/quality/history/
├── <commit>.json
├── <commit>.md
├── latest.json
└── latest.md
```

`json` は機械集計用、`md` はレビュー用である。working tree が dirty な状態で記録した場合は `<commit>-dirty.*` として保存し、レポート内にも `dirty: true` を残す。

## 17.7 CI / ローカル運用

- PR / main push では `.github/workflows/verify.yml` が `scripts/verify.sh` を実行する
- ローカルで変更前後の品質を見たい場合は `cargo run --quiet --example quality_score -- --min-score 90` を実行する
- コミットごとのスコア履歴を残す場合は `scripts/record-quality-score.sh` を実行し、生成された `docs/quality/history/` の差分をコミットに含める
