# 16. 設計判断

## 対象を md に限定した理由
ソースコードの索引化は `grep` / `ripgrep` / LSP で十分に機能しており、専用ツールとして抱える価値が薄い。一方で、Markdown ノートの増加に対する wiki 化（タグ・見出し・リンク索引）は、既存のエディタ依存ツール（Obsidian 等）から離れると途端に手段が乏しくなる。そこで v1 は「**自分で書いた md をそのまま育てる**」の 1 点に集中する。

## ノートを断片化する理由
v1 の出力は「AI に食わせて読解補助にする」ことを想定している。1 ノートを 1 ファイルにまとめると、当該ノートが長くなったときに読み込み単位が肥大化し、文脈としての密度が下がる。

そこで h2 を境界に自動分割し、小さな断片が互いにリンクしあう Zettelkasten 風の構造を採る。AI に対しては「関係する断片だけを注入する」こと、人間に対しては「単体でも読めるが隣接断片にすぐ跳べる」ことの双方を狙う。

## 断片の境界を h2 にした理由
h1 はノートタイトル相当で 1 ノートあたり 1 個が一般的なため、分割境界には使えない。h2 は「章」相当の粒度で、Markdown で自然に切れる場所である。ユーザが手で h2 を増やせば、そのまま分割も細かくなる。

h3 まで常に分割しない理由は、ほとんどの h3 は「同じ主張の一部分」に過ぎず、分割すると文脈が切れるから。ただし h2 セクションが長大化し、かつ h3 が複数含まれている場合のみ例外的に h3 分割を許す（FR-05）。

## 入口ページを別出ししている理由
原本のプレフェイス（h1 と最初の h2 までの前置き）には「そのノートが何を扱うか」の総論が書かれていることが多い。ここは分割せず、入口ページ (`fragments/<rel>/index.md`) に常に置く。入口ページはタグ索引や wikilink `[[NoteName]]` の着地点としても機能する。

## Parent / Prev / Next を冒頭に置いた理由
断片ページ単体を AI に渡したときに、最低限「親ページと前後断片がどこにあるか」が 1 行で読めるようにするため。ページ末尾ではなく冒頭に置くのは、LLM のコンテキスト先頭の方が構造情報として機能しやすいことと、人間が読むときも「この断片の立ち位置」が先に分かる方が自然なため。

## Backlinks を断片解像度にし、Related は入口だけに付けた理由
Backlinks は「どの断片から参照されているか」が分かると、AI が関連コンテキストを引っ張るときの粒度の細かいヒントになる。ノート単位で集約すると情報が潰れる。

一方で Related は「同じタグを持つ別ノート」レベルの緩い関連で、ノート単位のタグ集合からしか計算しない。これを全断片ページに複製するとノイズが増え、しかも全断片で内容が同一になるため、入口ページ 1 箇所に寄せる方が情報密度が高い。

## トップレベルを `fragments/` `tags/` `headings/` `links/` に平置きした理由
ディレクトリ名だけで役割を判断できることを優先した。役割ごとに命名規則が揃い（複数形のディレクトリ）、将来の拡張（例: `journal/`, `backlinks/`）も同じ形で追加しやすい。

## 見出し索引とリンク索引を分けた理由
- 見出し索引は「何が書かれているか」の目次
- リンク索引は「ページ間の参照関係」の地図

両者は役割が異なるため別ページに分割する。統合すると片方のユースケースがノイズになる。

## Obsidian 互換の `[[wikilink]]` を採用した理由
既存のノート資産を持つユーザがそのまま持ち込めること、またリンクの書き味が軽いこと。ただし `![[embed]]` は本文展開が原本を汚すため v1 では未対応（プレーンリンクに縮退）。

## AI 非依存を維持する理由
- API コストがかかると再生成が躊躇される
- 自然文要約を挟むと冪等性が壊れる
- 成果物はプレーンな Markdown のみ／ローカルのみで完結する、を譲らない

## `add` でも全体整合性を再計算する理由
索引・Backlinks・Related・未解決リンクは 1 ノートの追加でも全体に影響する。`add` は入力全体から desired output を作り、manifest と比較して filesystem への反映だけを差分化することで、増築操作と出力の整合性を両立する。

## 本文内 `#tag` を解釈しない理由
コードスニペット中の `#include` やシバン `#!/usr/bin/env` などとの誤検出対策をするとロジックが太る。フロントマター `tags` に集約することで扱いを単純化する。将来拡張候補（[§14](14-将来拡張.md)）。

## Retrieval compiler として再定義した理由
小さいコンテキストの agent では、生成 wiki 全体を読むことも、巨大な root index を読むことも現実的ではない。必要なのは、人間向け wiki そのものではなく、agent が目的に応じて根拠断片を集められる compile artifacts と context pack である。

## 静的 artifacts と retrieval CLI を両方持つ理由
静的 Markdown artifacts は透明性、レビュー性、デバッグ性を提供する。一方、agent が毎回 file tool だけで探索すると手順が長くなり、小さい LLM ほど失敗しやすい。`context` CLI は artifacts と internal catalog を使って、決定的な context pack を短い手順で返す。

## schema pack を外部 YAML にした理由
ゲーム、行政、業務などの domain field は Rust 本体に固定すべきではない。外部 YAML により、core engine は field 抽出と context recipe 実行に集中できる。

## `.md-wiki/catalog.json` を許容する理由
生成 wiki の可読成果物は Markdown を基本とするが、`context` CLI が毎回全 Markdown を読み直すと遅く不安定になる。内部 retrieval 用に JSON catalog を持つことで、速度、決定性、検証性を確保する。

## 意味推論をしない理由
`md-wiki` が「これは伏線である」「この関係は親子に近い」と推論すると、決定性と説明可能性が崩れる。未明示の意味判断は外部 agent / LLM に任せ、`md-wiki` は明示された field と source trail を提供する。
